When running the program, the output was:

Element 0: 1
Element 1: 2
Element 2: 3
Element 3: 4
Element 4: 5
Element 5: 0

The array only contains 5 elements:

int numbers[] = {1, 2, 3, 4, 5};

Valid indices are:

0 1 2 3 4

However, the program accessed Element 5, which is out of bounds.

🔎 Debugging Process
1️⃣ Launched GDB
gdb ./buggy_program
2️⃣ Set breakpoint at main
break main
run
3️⃣ Stepped through execution
next

The unexpected output appeared after calling:

buggy_function(numbers, 5);
🧠 Root Cause

Inside buggy_function, the loop condition was:

for (int i = 0; i <= size; i++)

Because size was passed as 5, the loop executed for:

i = 0,1,2,3,4,5

This caused access to:

arr[5]   // ❌ Out-of-bounds access

This is a classic off-by-one error.

Although the program did not crash, this behavior is undefined behavior in C and can lead to:

Segmentation faults

Memory corruption

Security vulnerabilities

✅ Solution

The loop condition was corrected from:

i <= size

to:

i < size
✔️ Corrected Code
void buggy_function(int arr[], int size) {
    for (int i = 0; i < size; i++) {
        printf("Element %d: %d\n", i, arr[i]);
    }
}
🧪 Result After Fix

Program output:

Element 0: 1
Element 1: 2
Element 2: 3
Element 3: 4
Element 4: 5

The invalid Element 5 output is no longer present.

📚 Lessons Learned

Always use < size when iterating over arrays.

Off-by-one errors are extremely common in C.

Undefined behavior may not immediately crash a program.

Compiling with -g and using GDB significantly simplifies debugging.

🏁 Conclusion

The bug was caused by an incorrect loop boundary condition.
After correcting the loop condition, the program behaves as expected.

This debugging session demonstrates the importance of:

Careful loop boundary design

Using debugging tools effectively

Understanding how array indexing works in C
